트랜잭션에 대한 작업 보고

transactionEventReport는 새로운 트랜잭션 이벤트를 보고하는 데 사용됩니다. 새롭게 생성된 이벤트는 트랜잭션 금액을 재계산하는 데 사용됩니다. 이 뮤테이션은 요청된 작업을 비동기적으로 처리하거나 결제 제공자 측에서 발생한 변경 사항을 보고하는 경우에 사용해야 합니다.

다음 인수를 사용합니다.

  • id: 트랜잭션의 ID입니다.
  • type: 보고된 작업의 유형입니다.
  • amount: 보고된 작업의 금액입니다.
  • pspReference: 작업에 할당된 참조입니다.
  • time: 작업의 시간입니다.
  • externalUrl: 작업 세부 정보가 있는 결제 제공자 페이지로 사용자를 리디렉션할 수 있는 URL입니다.
  • message: 작업과 관련된 메시지입니다.
  • availableActions: 트랜잭션에 대해 사용 가능한 작업 목록입니다.

transactionEventReport는 트랜잭션의 소유자만 호출할 수 있습니다. 소유자는 주어진 트랜잭션을 생성한 사용자 또는 앱입니다.

다음 예제는 transactionEventReport 뮤테이션을 사용하여 주어진 트랜잭션에 대해 발생한 이벤트를 보고하는 방법을 보여줍니다. 보고서는 성공적인 충전 작업이며, 금액은 20입니다. 통화는 트랜잭션에 선언된 것과 동일합니다. 트랜잭션에 대해 진행할 수 있는 사용 가능한 작업은 REFUND입니다. 제공된 데이터는 재계산 과정에 포함될 새로운 TransactionEvent 객체를 생성하는 데 사용됩니다.

mutation TransactionEventReport {
  transactionEventReport(
    id: "VHJhbnNhY3Rpb25JdGVtOjE="
    type: CHARGE_SUCCESS
    amount: 20
    pspReference: "psp-123"
    time: "2022-01-01"
    externalUrl: "https://saleor.io/event-details/123"
    message: "Charge completed"
    availableActions: [REFUND]
  ) {
    errors {
      field
      code
    }
    alreadyProcessed
    transaction {
      id
    }
    transactionEvent {
      id
    }
  }
}

응답에서 Saleor는 다음을 반환합니다:

  • alreadyProcessesed - 보고된 이벤트가 이전에 처리되지 않았음을 정의합니다. 입력 뮤테이션에서 제공된 것과 동일한 pspReference, 금액 및 유형을 가진 이벤트가 있으면 Saleor는 새로운 것을 생성하는 대신 그것을 반환하고 플래그가 true로 설정됩니다. 제공된 pspReference 및 유형이 이미 보고되었지만 다른 금액으로 처리된 경우, 코드가 INCORRECT_DETAILS인 오류가 발생합니다.
  • transaction - 수신된 보고서를 기반으로 업데이트된 트랜잭션입니다.
  • transactionEvent - 수신된 보고서를 기반으로 생성된 TransactionEvent입니다.

transactionRequestAction을 호출하면 요청 유형 중 하나(AUTHORIZATION_REQUEST, CHARGE_REQUEST, REFUND_REQUEST, CANCEL_REQUEST), 금액, 소유자(User 또는 App)로 새로운 TransactionEvent가 생성됩니다. Saleor는 TRANSACTION_CHARGE_REQUESTED, TRANSACTION_CANCELATION_REQUESTED, TRANSACTION_REFUND_REQUESTED에 대한 동기 웹훅을 전송합니다.

응답에는 적어도 액션의 pspReference가 포함되어야 합니다. pspReference는 이전에 생성된 _REQUEST 유형의 이벤트에 배치됩니다. 선택적으로 응답에 완료된 액션의 세부 정보를 포함할 수 있습니다.

비동기식 액션 처리

액션이 결제 제공 업체 측에서 비동기식으로 처리되는 경우, 앱은 결제 제공 업체로부터 웹훅 알림을 받으면 transactionActionRequest 변이를 호출해야 합니다.

동기식 액션 처리

앱은 요청된 액션의 상태를 즉시 받습니다. 앱은 수신한 Saleor 웹훅에 대한 응답으로 액션의 세부 정보를 제공할 수 있습니다. 다음 웹훅 이벤트는 응답에 액션 세부 정보를 수용할 수 있습니다: TRANSACTION_CHARGE_REQUESTED, TRANSACTION_CANCELATION_REQUESTED, TRANSACTION_REFUND_REQUESTED.

앱이 Saleor 대시보드 측의 직원 사용자 또는 앱에 의해 트리거된 결제 요청 액션(충전, 환불 또는 취소와 같은)도 처리해야 한다면, 앱이 transaction-action-request 웹훅을 구독하도록 확인하세요.

거래 금액의 재계산

재계산은 transactionEventReport, transactionInitialize, transactionProcess 뮤테이션에서 제공되는 TransactionEventTypeEnum의 값에 따라 다릅니다. 웹훅 응답에서 선택적으로 사용할 수 있습니다:

이벤트 유형 설명

AUTHORIZATION_SUCCESS

Saleor는 제공된 금액을 transaction.authorizedAmount에 할당합니다. AUTHORIZATION_SUCCESS 이벤트가 pspReference에 의해 AUTHORIZATION_REQUEST 이벤트와 관련이 있다면, transactionItem.authorizePendingAmount는 AUTHORIZATION_SUCCESS 이벤트에서 보고된 금액만큼 감소됩니다. 단일 거래에 대해 두 개 이상의 성공적인 인증을 가질 수 없습니다. 첫 번째 요청 이후의 모든 성공 요청은 예외를 발생시킵니다. AUTHORIZATION_ADJUSTMENT은 transaction.authorizedAmount를 변경하는 데 사용해야 합니다

AUTHORIZATION_SUCCESS와 AUTHORIZATION_FAILURE가 동일한 pspReference를 가진 경우를 발견할 경우, transaction.authorizedAmount의 재계산은 AUTHORIZATION_SUCCESS가 이전 것일 때 AUTHORIZATION_SUCCESS의 금액을 고려하지 않습니다.

AUTHORIZATION_FAILURE

AUTHORIZATION_FAILURE 이벤트가 pspReference에 의해 AUTHORIZATION_REQUEST 이벤트와 관련이 있다면, transactionItem.authorizePendingAmount는 AUTHORIZATION_FAILURE 이벤트에서 보고된 금액만큼 감소됩니다. 관련된 AUTHORIZATION_REQUEST가 없으면 실패 이벤트는 transactionItem 액션의 기록을 제공하는 데만 사용됩니다.

AUTHORIZATION_ADJUSTMENT

Saleor는 제공된 금액을 transaction.authorizedAmount에 할당합니다.

CHARGE_SUCCESS

제공된 금액을 transaction.chargedAmount에 추가합니다. 다음 호출마다 기존 transaction.chargedAmount에 금액이 추가됩니다 (transaction.chargedAmount += amount). CHARGE_SUCCESS 이벤트가 pspReference에 의해 CHARGE_REQUEST 이벤트와 관련이 있다면, transactionItem.chargePendingAmount는 CHARGE_SUCCESS 이벤트에서 보고된 금액만큼 감소됩니다. 관련된 CHARGE_REQUEST 이벤트가 없으면 제공된 금액이 transaction.authorizedAmount를 줄입니다.

CHARGE_SUCCESS와 CHARGE_FAILURE가 동일한 pspReference를 가진 경우를 발견할 경우, transaction.chargedAmount의 재계산은 CHARGE_SUCCESS가 이전 것일 때 CHARGE_SUCCESS의 금액을 고려하지 않습니다.

CHARGE_FAILURE

CHARGE_FAILURE 이벤트가 pspReference에 의해 CHARGE_REQUEST 이벤트와 관련이 있다면, transactionItem.chargePendingAmount는 CHARGE_FAILURE 이벤트에서 보고된 금액만큼 감소됩니다. 관련된 CHARGE_REQUEST가 없으면 실패 이벤트는 transactionItem 액션의 기록을 제공하는 데만 사용됩니다.

CHARGE_BACK

제공된 금액은 transaction.chargedAmount를 줄이는 데 사용됩니다 (transaction.chargedAmount -= amount).

REFUND_SUCCESS

제공된 금액을 transaction.refundedAmount에 추가합니다. 다음 호출마다 기존 transaction.refundedAmount에 금액이 추가됩니다 (transaction.refundedAmount += amount). REFUND_SUCCESS 이벤트가 pspReference에 의해 REFUND_REQUEST 이벤트와 관련이 있다면, transactionItem.refundPendingAmount는 REFUND_SUCCESS 이벤트에서 보고된 금액만큼 감소됩니다. 관련된 REFUND_REQUEST 이벤트가 없으면 제공된 금액이 transaction.chargedAmount를 줄입니다.

REFUND_SUCCESS와 REFUND_FAILURE가 동일한 pspReference를 가진 경우를 발견할 경우, transaction.refundedAmount의 재계산은 REFUND_SUCCESS가 이전 것일 때 REFUND_SUCCESS의 금액을 고려하지 않습니다.

REFUND_FAILURE

REFUND_FAILURE 이벤트가 pspReference에 의해 REFUND_REQUEST 이벤트와 관련이 있다면, transactionItem.refundPendingAmount는 REFUND_FAILURE 이벤트에서 보고된 금액만큼 감소됩니다. 관련된 REFUND_REQUEST가 없으면 실패 이벤트는 transactionItem 액션의 기록을 제공하는 데만 사용됩니다.

REFUND_REVERSE

제공된 금액은 transaction.refundedAmount를 줄이는 데 사용됩니다 (transaction.refundedAmount -= amount). 제공된 금액은 transaction.chargedAmount를 증가시킵니다.

CANCEL_SUCCESS

제공된 금액이 transaction.canceledAmount에 추가됩니다 (transaction.canceledAmount += amount). CANCEL_SUCCESS 이벤트가 pspReference에 의해 CANCEL_REQUEST 이벤트와 관련이 있다면, transactionItem.cancelPendingAmount는 CANCEL_SUCCESS 이벤트에서 보고된 금액만큼 감소됩니다. 관련된 CANCEL_REQUEST 이벤트가 없으면 제공된 금액이 transaction.authorizedAmount를 줄입니다.

CANCEL_SUCCESS와 CANCEL_FAILURE가 동일한 pspReference를 가진 경우를 발견할 경우, transaction.canceledAmount의 재계산은 CANCEL_SUCCESS가 이전 것일 때 CANCEL_SUCCESS의 금액을 고려하지 않습니다.

CANCEL_FAILURE

CANCEL_FAILURE 이벤트가 pspReference에 의해 CANCEL_REQUEST 이벤트와 관련이 있다면, transactionItem.cancelPendingAmount는 CANCEL_FAILURE 이벤트에서 보고된 금액만큼 감소됩니다. 관련된 CANCEL_REQUEST가 없으면 실패 이벤트는 transactionItem 액션의 기록을 제공하는 데만 사용됩니다.
